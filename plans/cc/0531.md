  数据集 CSV → 归一化处理 → 5 个专家 Agent 各自预测 → Choquet 聚合层 → 最终分类

  Agent 后端三种模式

  ┌────────┬───────────────────────────────────────────┐
  │  模式  │                   行为                    │
  ├────────┼───────────────────────────────────────────┤
  │ rule   │ 关键词匹配 + 手工打分公式，不需要 API key │
  ├────────┼───────────────────────────────────────────┤
  │ hybrid │ 优先 LLM，失败时回退 rule                 │
  ├────────┼───────────────────────────────────────────┤
  │ llm    │ 纯 LLM，失败则报错退出                    │
  └────────┴───────────────────────────────────────────┘

  Choquet 聚合层两种模式

  1. inspired：单 Agent 权重（softmax） + 两两交互权重（sigmoid），公式为 score = Σ w_i × p_i + λ × mean_{i<j}(w_ij ×
    p_i × p_j)
  2. discrete_2additive：严格 Choquet 积分的 2-additive Möbius 近似，C_μ(f) = Σ [f_σ(i) - f_σ(i-1)] × μ({σ(i),...,σ(K)})

  ├─────────────────────────────┼────────────────────┼────────────┼──────────┤

  └─────────────────────────────┴────────────────────┴────────────┴──────────┘

  核心结论：规则 Agent 已经达到 100% 准确率，而 LLM Agent 只有 ~54%（接近随机水平）。LLM Agent 是整个系统的瓶颈。

---
  为什么 LLM Agent 表现这么差？深层原因分析

  1. 规则 Agent 已经完美，说明数据集的模式是高度规则化的

  zongxiang.csv 数据集（2120 行）中，标题党 vs 非标题党有明显的词汇信号（如"震惊"、"秘密"、"万万没想到" vs
  "报告"、"预算"、"公布"）。规则 Agent 的手工关键词列表恰好覆盖了这些模式，所以达到 100%。

  2. LLM Agent 的致命缺陷

  - 单一模型 × 5 个角色 = 伪多样性：5 个 LLM Agent 底层都是同一个 gemini-3.1-flash-lite 模型，只是 system prompt
    不同。实际效果是同一模型思考 5 次，没有真正的模型集成多样性。
  - gemini-3.1-flash-lite 能力不足：这是一个低成本小模型，对于中文标题党检测这种需要细腻判断的任务可能力不从心。
  - LLM 可能过度推理：规则能直接匹配"震惊"=标题党，但 LLM 可能被复杂的 prompt
    引导去考虑过多因素，反而失去了直接判断能力。

  3. Choquet 聚合层无法修复错误的 Agent 输入

  Choqet 聚合层学的是"如何组合 agent 输出"，但如果所有 agent 输出都是噪声（~50%
  准确率），那学到的权重再好也没用。这就像让一个裁判去裁判 5 个盲人的判断。

---
  改进建议（按优先级排序）

  🔴 高优先级（大幅提升准确率）

  1. 多模型集成（Ensemble）

  当前 config.py 已定义了 LLM_MODEL_CANDIDATES 列表但只用了一个模型。让 5 个 Agent 使用不同的底层模型：
  # 改进方案：每个 agent 分配不同的模型
  agent_model_map = {
      "Semantic": "gemini-2.5-flash",       # 推理能力强的模型做语义
      "Emotion": "gemini-3.1-flash",        # 中等模型做情绪
      "Intention": "gemini-2.0-flash",      # 不同模型增加多样性
      "Lexical": "gemini-3.1-flash-lite",   # 词汇层面用轻量模型即可
      "Consistency": "gemini-2.5-flash-lite", # 又一个不同模型
  }
  这能引入真正的模型层面多样化，ensemble 效果会显著提升。

  2. 增加规则 Agent 作为固定锚点

  即使使用 LLM 模式，也应该保留规则 Agent 作为 baseline anchor。让 LLM Agent 和 Rule Agent 共同组成 agent pool：
  Agent Pool = [RuleAgents × 5, LLMAgents × 5] → 10 个 agent → Choquet 聚合
  规则 Agent 已经 100% 准确，它们会为 Choquet 聚合提供强信号支撑。

  3. 改进 LLM Prompt 策略

  当前 prompt 要求 LLM 输出概率值，但 LLM 的概率校准很差。改为：
  - 先判断再打分：让 LLM 先输出"是/否"的定性判断，再基于判断理由给出置信度
  - 增加 Few-shot 示例：在 system prompt 中加入 3-5 个标注好的示例，引导 LLM 理解任务边界
  - 结构化输出：使用更严格的 JSON Schema 约束

  4. 升级 LLM 模型

  gemini-3.1-flash-lite → gemini-2.5-flash 或 gemini-3.1-flash。LLM_MODEL_CANDIDATES 已经列出了更好的模型选项。

  🟡 中优先级（优化模型架构）

  5. Agent 权重置信度门控

  当 agent 置信度低时自动降低其权重：
  # 在 Choquet 层加入 confidence gating
  gated_probs = agent_probs * (agent_confidences > threshold).float()

  6. Agent 间一致性特征

  增加一个新特征：每个 agent 与其他 agent 的一致性得分，帮助 Choquet 层识别"异类" agent：
  consistency_i = mean(cosine_similarity(agent_i_probs, agent_j_probs) for j ≠ i)

  7. 数据增强

  - 当前规则 Agent 给了 perfect 结果，说明关键词特征很强。可以基于规则 Agent
    发现的关键词做同义词替换数据增强，让模型不只依赖特定词汇。
  - 用 LLM 生成更多变体文本（paraphrasing），增加数据多样性。

  🟢 低优先级（工程优化）

  8. 训练策略改进

  - 当前 35 epoch + lr=0.01，对于 Choquet 层可能过多。加入 early stopping（patience=10）
  - 添加 学习率调度器（ReduceLROnPlateau）
  - 在 discrete_2additive 模式下加入容量单调性正则化损失，让学到的 capacity 更符合数学定义

  9. 特征工程增强

  - 当前只用 char n-gram TF-IDF。可以加入：
    - BGE/M3E 等中文 embedding 模型替代 TF-IDF
    - 文本长度、标点密度等统计特征
    - 标题与内容的一致性特征

  10. 缓存策略优化

  当前缓存 key 包含完整文本，稍微改一个字就会 cache miss。可以考虑对文本做归一化（去空格、统一标点）后再生成 key。

---
  总结

  当前最有效的改进路径：

  1. 短期（立刻有效）：用 AGENT_BACKEND=rule 即可获得 100% 准确率。这个数据集的规则模式已经被完美覆盖。
  2. 中期（提升 LLM 模式）：实施多模型集成 + 混合 rule/LLM agent pool，预期可将 LLM 模式准确率从 54% 提升到 80%+。
  3. 长期（泛化能力）：增强 prompt 工程 + 数据增强 + 更好的 embedding，使模型在规则覆盖不到的数据上也能表现良好。

  核心洞察：问题不在 Choquet 聚合层，而在底层 Agent 的信号质量。 规则 Agent 已经证明了 100% 的可行性，说明 Choquet
  架构本身是正确的。提升 LLM Agent 的信号质量才是提高正确率的关键。
